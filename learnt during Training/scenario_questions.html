S1–S10
S1
🔥 Most Common
"Your Selenium test is failing intermittently — it passes 7 out of 10 times. What would you do to investigate and fix it?"
▼

Why they ask this

Intermittent failures are one of the biggest real-world challenges in automation. They want to see if you understand synchronization, stability, and systematic debugging.

How I would answer

1
First — check for timing/wait issues. Intermittent = usually a synchronization problem. The page loads slowly sometimes. I'd look at every findElement() call and ensure there's an explicit wait before each critical interaction: wait.until(ExpectedConditions.elementToBeClickable(...))
2
Remove all Thread.sleep(). These are fixed waits — if the page takes 4 seconds but sleep is 3s, it fails. Replace every sleep with WebDriverWait.
3
Check for StaleElementReferenceException in logs. If AJAX refreshes the DOM, cached element references go stale. Fix: find the element fresh inside a retry loop or after each navigation.
4
Take screenshot on failure (if not already). The screenshot shows exactly what the browser looked like at the moment of failure — is a popup blocking? Is the page still loading? Did the element shift position?
5
Add more detailed logging — log each action with timestamp. Compare logs from pass run vs fail run to spot where timing differs.
6
Check test independence — is this test relying on state left by a previous test? Tests should be independent. If test A leaves browser in unexpected state, test B intermittently fails.
One-line answer: Intermittent failures = almost always a wait/synchronization issue. Replace Thread.sleep with WebDriverWait, check for StaleElement, take screenshots on failure, and ensure tests are independent.
S2
🔥 Very Common
"Your Selenium test cannot find the element. NoSuchElementException is thrown. Walk me through how you debug this."
▼

Why they ask this

This happens daily in real projects. They want to see your systematic debugging approach.

How I would answer — my debugging checklist

1
Verify the locator in browser DevTools first. Open the page manually, press F12, go to Console tab, and test: document.getElementById('username') or $x("//input[@id='username']"). If it returns null, the locator is wrong.
2
Is the element inside an iFrame? If yes, I forgot to switch: driver.switchTo().frame("frameId") before finding the element.
3
Is the element dynamically loaded? Page may have loaded but element not yet appeared (AJAX). Add: wait.until(ExpectedConditions.presenceOfElementLocated(By.id("username")))
4
Is the element inside Shadow DOM? Normal locators can't see inside Shadow DOM. Use: host.getShadowRoot().findElement(By.cssSelector("input"))
5
Did the page navigate correctly? Print driver.getCurrentUrl() and driver.getTitle() before the failing line to confirm you're on the right page.
6
Is the element in a different window/tab? A popup may have opened. Check driver.getWindowHandles().size() and switch if needed.
Answer: I first verify the locator in DevTools console. Then check: is element in iframe? Not yet loaded (need wait)? In Shadow DOM? Wrong page? Wrong window? Systematic elimination.
S3
Common
"How would you automate a scenario where you need to login once and run 20 different test cases without logging in again for each test?"
▼

Why they ask this

Tests that login before every test case are slow. Real projects optimize this. They want to see if you know @BeforeClass, cookie handling, and test design.

How I would answer — 3 approaches

1
Approach 1 — @BeforeClass (simplest). Login once in @BeforeClass. All 20 @Test methods share the same browser session. @AfterClass logs out and closes browser. Works when tests don't log out or need fresh login.
2
Approach 2 — Cookie-based login (fastest). Login once, capture the session cookie: Cookie sessionCookie = driver.manage().getCookieNamed("sessionId"). Store it. For next test: navigate to base URL, add the cookie, refresh — you're logged in instantly without going through login page.
3
Approach 3 — API login (fastest + most reliable). Instead of UI login, send a POST request to login API endpoint using REST Assured, get the auth token, add it to browser cookies or local storage. Zero UI interaction for login.
// @BeforeClass approach @BeforeClass public void loginOnce() { driver = new ChromeDriver(); driver.get("https://app.com/login"); new LoginPage(driver).login("admin", "pass"); // All 20 @Test methods share this session }
Answer: Use @BeforeClass to login once before all tests. For even faster approach, save session cookie after login and reuse it — skipping login UI entirely for subsequent tests.
S4
🔥 Common
"You click a button and get ElementClickInterceptedException. The element exists and is visible. What happened and how do you fix it?"
▼

Why they ask this

This is extremely common on modern websites with overlays, cookie banners, and modals. Tests fail unexpectedly.

How I would answer

1
What happened: Another element is physically on top of the target element — intercepting the click. Common culprits: cookie consent banner, "Accept all cookies" popup, loading spinner overlay, sticky header, modal dialog, GDPR notice.
2
Fix 1 — Dismiss the overlay first: Wait for and close the cookie banner/popup before clicking the target: wait.until(ExpectedConditions.elementToBeClickable(By.id("acceptCookies"))).click()
3
Fix 2 — Scroll element into view first: ((JavascriptExecutor) driver).executeScript("arguments[0].scrollIntoView(true);", element) then click normally.
4
Fix 3 — JavaScript click (last resort): ((JavascriptExecutor) driver).executeScript("arguments[0].click();", element) — fires click directly on DOM, bypasses any overlay. Used in my Rediff project for Terms link.
⚠️ Always try Fix 1 or Fix 2 first. JS click bypasses real user behavior and may hide genuine UI issues. Use it only when other approaches don't work.
Answer: An overlay is covering the element. I first try to close the overlay, then scroll element into view. If still failing, use JavascriptExecutor to click — which fires directly on the DOM element bypassing any intercepting layer.

Common
"How would you automate testing of a date picker calendar widget? It uses a JavaScript calendar popup."
▼

Why they ask this

Date pickers are notoriously tricky to automate. They want to see if you know multiple approaches.

How I would answer — 3 approaches

1
Approach 1 — Direct input (if allowed): If the date field is a text input, just use sendKeys("12/25/2024"). Fastest. Check input type — if it's type="date" use format "2024-12-25".
2
Approach 2 — Interact with calendar UI: Click the calendar icon → wait for calendar popup → click next/prev arrows to navigate to correct month → click the date cell. Uses Actions class for complex navigation.
3
Approach 3 — JavaScript to set value directly: js.executeScript("document.getElementById('dateField').value='2024-12-25';") — bypasses the calendar UI entirely. Fastest but doesn't test the calendar UI itself.
4
For the calendar UI approach, I'd use Actions class: actions.moveToElement(nextBtn).click() to navigate months. For each date cell, build XPath: //td[@data-date='25'] and click.
Answer: I prefer sendKeys directly to the date input field — it's fast and reliable. If the field is read-only, I use JavaScript to set the value. Only use calendar UI interaction if specifically testing the calendar widget itself.
S6
Common
"Your test passes locally on Chrome but fails on the CI server (Jenkins/Linux). What are the possible reasons and how would you fix it?"
▼

Why they ask this

CI/CD is used in every company. This scenario tests your knowledge of headless browsers, environment differences, and CI setup.

How I would answer — common causes

1
No display on Linux server. Jenkins server has no GUI. Chrome can't open a visible window. Fix: run headless — options.addArguments("--headless=new"). Or use Xvfb (virtual framebuffer) on Linux.
2
Chrome version mismatch. Your local Chrome = v124, server Chrome = v118. ChromeDriver for v124 doesn't work with v118. Fix: use WebDriverManager — it auto-detects version and downloads correct driver.
3
Different screen resolution. Local = 1920×1080, server = 1024×768. Elements not visible or positioned differently at smaller resolution. Fix: options.addArguments("--window-size=1920,1080")
4
File path issues. Your test uses absolute Windows paths (C:\testdata\). Server is Linux (no C:\). Fix: use relative paths or System.getProperty("user.dir") for project root.
5
Shared memory (Docker). If running in Docker, --no-sandbox and --disable-dev-shm-usage flags needed to prevent Chrome crashes.
Answer: Most likely cause is no display on server — run headless. Also check Chrome version, screen resolution, file paths, and if in Docker add --no-sandbox flag. WebDriverManager handles driver version automatically.

Common
"There's a table on the page with 100 rows of employee data. How would you verify that a specific employee 'John Smith' exists and get his salary?"
▼

How I would answer

1
Approach 1 — Direct XPath (most efficient): Use text-based XPath to find the row containing "John Smith" then get sibling cell for salary: //td[text()='John Smith']/following-sibling::td[@class='salary']. No looping needed.
2
Approach 2 — Loop through rows: List<WebElement> rows = driver.findElements(By.cssSelector("table tbody tr")). For each row, get cells, check if first cell contains "John Smith", then get salary cell.
3
Verify existence: Use findElements() (returns empty list instead of exception if not found): Assert.assertTrue(driver.findElements(By.xpath("//td[text()='John Smith']")).size() > 0, "Employee not found")
// Most elegant — direct XPath String salary = driver.findElement( By.xpath("//td[text()='John Smith']/following-sibling::td[2]") ).getText(); Assert.assertEquals(salary, "50000");
Answer: I use XPath with text() to directly locate the row containing "John Smith", then use following-sibling:: axis to get the salary column — no looping needed, efficient for any table size.

Common
"How would you handle a scenario where you need to test the same user flow on Chrome, Firefox, and Edge?"
▼

How I would answer — Cross Browser Testing

1
DriverFactory pattern: Create a factory class that takes a browser name string and returns the appropriate driver. Test code never creates a driver directly — calls DriverFactory.getDriver("chrome").
2
testng.xml parameterization: Create 3 <test> blocks each passing different browser parameter. Run with parallel="tests" to run all 3 simultaneously. @Parameters({"browser"}) in @BeforeClass receives it.
3
Selenium Grid for true parallel: Hub + Chrome Node + Firefox Node + Edge Node. Each browser test sent to appropriate node. Saves time — runs all 3 browsers simultaneously.
4
config.properties approach: Read browser from config file. Change one line in config → all tests switch browser. Jenkins can pass browser as parameter: mvn test -Dbrowser=firefox
// DriverFactory public static WebDriver getDriver(String browser) { switch(browser.toLowerCase()) { case "chrome": WebDriverManager.chromedriver().setup(); return new ChromeDriver(); case "firefox": WebDriverManager.firefoxdriver().setup(); return new FirefoxDriver(); case "edge": WebDriverManager.edgedriver().setup(); return new EdgeDriver(); default: throw new IllegalArgumentException("Browser not supported: " + browser); } }
Answer: DriverFactory pattern with switch-case + testng.xml parameterization for browser name + Selenium Grid for parallel execution across browsers. Read browser from config.properties so CI/CD can override easily.

Common
"How would you handle file upload in Selenium? What if the file upload dialog is a native OS window that Selenium can't control?"
▼

How I would answer

1
Approach 1 — sendKeys on file input (most common): If it's an HTML <input type="file">, just use sendKeys with the full file path — no need to click! driver.findElement(By.id("fileInput")).sendKeys("C:/files/test.pdf"). Selenium bypasses the dialog entirely.
2
Approach 2 — Robot class (for native OS dialogs): If clicking a button opens a Windows/Mac file dialog (native OS popup Selenium can't handle), use Java's Robot class to type the file path and press Enter: robot.keyPress(KeyEvent.VK_ENTER). Works on local machine, unreliable on headless.
3
Approach 3 — AutoIT (Windows only): AutoIT is a scripting tool that can handle native Windows dialogs. Compile a script that enters the file path and call it from Java using Runtime.getRuntime().exec().
4
Best practice: Design ChromeOptions to auto-set the download/upload directory: prefs.put("download.default_directory", "/path/to/files") so file dialogs auto-populate.
Answer: For HTML file inputs, sendKeys with the file path is all I need — Selenium sets the value directly without opening any dialog. For native OS dialogs, I use Robot class or AutoIT to interact with the OS-level window.

"You join a new project with no automation in place. The team does everything manually. How would you set up a Selenium automation framework from scratch?"
▼

How I would answer — My step-by-step plan

1
Understand the application first. Explore the app manually. Identify high-value, frequently used features. Talk to business team about critical user journeys. These become first automation candidates.
2
Set up Maven project in IntelliJ. Add dependencies: Selenium, TestNG, WebDriverManager, Apache POI, ExtentReports in pom.xml. Create proper folder structure: pages/, tests/, utils/, base/.
3
Build BaseTest class first. Common @BeforeMethod (driver setup, maximize, navigate) and @AfterMethod (quit). All test classes extend this — no repetition.
4
Create Page classes for each page using POM + PageFactory. Private @FindBy locators, public action methods. Enables test code that reads like English: loginPage.login(user, pass).
5
Add config.properties for browser, URL, credentials. ConfigReader utility class. Switching environments = change one config file.
6
Integrate reporting: ExtentReports for beautiful HTML reports. TestNG Listeners for automatic screenshot on failure. Reports tell the story of each test run.
7
Connect to CI/CD: Push to GitHub. Create Jenkins job with mvn clean test. Set up to run nightly and on every PR merge.
8
Start with smoke suite (5-10 critical tests). Prove value quickly. Then expand to regression suite (25+ tests). Don't try to automate everything at once.
Answer: Maven + TestNG + Selenium + POM + PageFactory + BaseTest + ConfigReader + ExtentReports + Jenkins. Start small with smoke tests, prove value, then expand. Always prioritize high-business-value scenarios first.

"How would you design your framework to handle test data — some tests need different data, some need data-driven testing from Excel?"

How I would answer — layered data strategy

1
Static data in config.properties: Fixed test data that rarely changes — base URL, valid credentials for login setup. admin.user=testadmin, admin.pass=Admin@123

Common
"50 test cases all need to login first. How do you avoid writing login code 50 times? How do you make this maintainable?"
▼

How I would answer — DRY principle

1
Login in BaseTest @BeforeMethod: Put login logic once in BaseTest. All 50 test classes extend BaseTest — they automatically get login before each test. Zero code duplication.
2
LoginPage as reusable POM class: Login logic is in LoginPage.login(user, pass). BaseTest creates LoginPage object and calls it. If login UI changes, update ONLY LoginPage class — all 50 tests automatically get the fix.
3
Credentials from config.properties: Username/password in config, not hardcoded. Change user = change one config line. Different environments (staging/prod) can have different credentials.
4
If 50 tests run in sequence on same browser (@BeforeClass): Login once, all 50 tests run, logout at end. Even more efficient — 1 login for 50 tests.
public class BaseTest { protected WebDriver driver; @BeforeMethod public void setUp() { driver = DriverFactory.getDriver(ConfigReader.get("browser")); driver.get(ConfigReader.get("url")); new LoginPage(driver).login( ConfigReader.get("username"), ConfigReader.get("password") ); } @AfterMethod public void tearDown() { driver.quit(); } }
Answer: BaseTest class + POM LoginPage + config.properties. Login once in @BeforeMethod, all tests extend BaseTest. If login changes, update ONE class. This is the DRY principle in action.

Common
"You have 200 automated tests. Full run takes 2 hours. Management wants it to run in 30 minutes. What strategies would you use?"
▼

How I would answer — parallelization + optimization

1
Parallel execution with TestNG: Set parallel="methods" thread-count="4" in testng.xml. Use ThreadLocal for driver. 200 tests ÷ 4 threads = ~30 mins. This alone can achieve 4x speedup.
2
Selenium Grid for distributed execution: Spread tests across 4-5 machines. Each machine runs a subset. Hub distributes automatically. 200 tests across 4 machines = effectively 50 tests per machine.
3
Run only critical tests in CI (smart subset): Tag tests by type. Run @smoke (20 tests, ~10 min) on every commit. Run full @regression (200 tests) only nightly.
4
Optimize slow tests: Profile tests — find the 20 slowest. Often it's unnecessary Thread.sleep() or loading pages that could be skipped. Replace sleeps with explicit waits.
5
Headless execution: Headless Chrome is ~20-30% faster than visible Chrome (no rendering overhead).
6
API for setup, not UI: Instead of clicking through UI to create test data, use API calls. Login via API + cookie. This saves 10-30 seconds per test.
Answer: Parallel execution (TestNG + ThreadLocal) + Selenium Grid + headless + smoke/regression split + replace Thread.sleep + API-level setup. Combined, easily achieves 4-5x speedup.

S17
Common
"You have a test that verifies checkout flow: add item → go to cart → enter payment → confirm order. If 'add item' fails, what happens to the rest? How would you design this?"
▼

How I would answer

1
Option A — Single test method (best for this flow): All steps in one @Test method. If step 1 fails, steps 2-4 don't run (test stops). Clean, simple, no confusion. Best for sequential user journeys where each step depends on previous.
2
Option B — dependsOnMethods: Each step as separate @Test with dependsOnMethods. If "addItem" fails → "goToCart" is skipped (not failed). Report shows: addItem=FAIL, goToCart=SKIP, enterPayment=SKIP. Clear chain of failure.
3
My recommendation: Single test for end-to-end flows. Separate tests for testing each feature independently. Don't mix — sequential flow = one test, independent feature checks = separate tests.
@Test(priority=1) public void addItemToCart() { ... } @Test(priority=2, dependsOnMethods="addItemToCart") public void proceedToCheckout() { ... } @Test(priority=3, dependsOnMethods="proceedToCheckout") public void enterPaymentDetails() { ... }
Answer: For sequential user flows, I use either a single test method with all steps, or separate @Test methods chained with dependsOnMethods — so downstream tests are SKIPPED (not failed) when an upstream step fails. This gives clear, actionable test reports.
S18
Common
"How would you verify that your test is testing exactly what you think it's testing — i.e., how do you ensure test reliability and avoid false positives?"
▼

How I would answer

1
Assert meaningfully — not just status. Don't just assert page loads. Assert specific expected values: Assert.assertEquals(driver.getTitle(), "Dashboard - MyApp"). A wrong page can also have "Dashboard" somewhere.
2
Test independence — clean state every time. Each test starts fresh (new browser or fresh logged-in state). Tests that share state cause false positives — test B passes only because test A left it in the right state.
3
Run tests in random order. TestNG can randomize order. If tests only pass in specific order, they have hidden dependencies — that's a bug in your tests.
4
Verify negative scenarios too. Test that invalid login DOES show an error. Test that a deleted item DOES NOT appear. False positives often come from not testing the negative.
5
Code review for tests. Just like production code, test code should be reviewed. Another person catches wrong assertions, missing edge cases, brittle locators.
Answer: Meaningful specific assertions, test independence, negative test cases, randomized execution order, and peer review of test code. A test that always passes is suspicious — make sure it can actually fail when it should.

S20
Common
"In your Rediff project, why did you use @BeforeClass instead of @BeforeMethod? What tradeoff did you make?"
▼

How I would answer — defend your project design

1
Why @BeforeClass: All 5 test cases are part of one sequential user journey on Rediffmail — homepage → registration page → links → Terms window → child window. They all share the same browser session intentionally.
2
Efficiency gain: @BeforeClass opens Chrome ONCE. @BeforeMethod would open Chrome 5 times and close 5 times. That's 5× the browser launch overhead. For time-sensitive CI pipelines, this matters.
3
The tradeoff I accepted: Tests are not 100% independent. TC02 depends on TC01 navigating to the homepage. I handled this with dependsOnMethods — if TC01 fails, subsequent tests skip cleanly rather than failing with confusing errors.
4
Also used singleThreaded=true: Since tests share one browser, they must run on the same thread sequentially. Without this, parallel execution would have multiple threads fighting over the same driver.
Answer: @BeforeClass for efficiency — 1 browser launch for 5 tests. Tradeoff: tests share state, but I mitigated this with dependsOnMethods for clean skip behavior and singleThreaded=true to prevent parallel thread conflicts.

"Two different feature files both need 'Given user is on the login page'. How do you avoid duplicating the step definition code?"

1
Step definitions are automatically shared. You write the step def ONCE in one class. Any feature file that uses the EXACT same text "Given user is on the login page" automatically maps to the same Java method. No duplication needed.

Common
"Your BDD tests take 2 hours to run. Management wants only critical tests run on every commit. How would you organize this?"
▼

How I would answer — tag-based execution strategy

1
Tag critical scenarios @smoke: 10-15 most important end-to-end scenarios. Login, core feature, checkout. Everything critical to business. These run in ~10 minutes.
2
@regression tag for all: Full 2-hour suite. Runs nightly or before releases — not on every commit.
3
Jenkins multi-pipeline strategy: Pipeline 1 (on every commit): tags = "@smoke". Pipeline 2 (nightly 10pm): tags = "@regression". Pipeline 3 (before release): tags = "not @skip".
4
In Runner class: tags = "@smoke" or override from Maven: mvn test -Dcucumber.filter.tags="@smoke". Jenkins passes the tag parameter per pipeline.
Answer: Tag scenarios with @smoke, @regression, @sanity. Run @smoke on every commit (~10 mins), @regression nightly (~2 hours). Two Jenkins pipelines with different tag filters. Fast feedback for developers, full coverage for releases.

"When would you choose BDD (Cucumber) over regular TestNG tests? When would you NOT use BDD?"
▼

How I would answer — honest assessment

1
Use BDD when: Business stakeholders actively participate in defining test scenarios. Team values living documentation. Non-technical clients want to read/verify test coverage. Feature specs are complex and need clear collaboration. Acceptance testing with clear Given/When/Then criteria.

"You need to test a complete user CRUD flow via API: create user → get user → update user → delete user → verify deleted. How would you write this in REST Assured?"

How I would answer — chained API tests

public class UserCrudTest { private static String userId; @Test(priority=1) public void createUser() { String body = "{ \"name\": \"John\", \"email\": \"john@test.com\" }"; Response res = given().contentType("application/json").body(body) .when().post("/users") .then().statusCode(201).extract().response(); userId = res.jsonPath().getString("id"); // save ID } @Test(priority=2, dependsOnMethods="createUser") public void getUser() { given().pathParam("id", userId) .when().get("/users/{id}") .then().statusCode(200).body("name", equalTo("John")); } @Test(priority=3, dependsOnMethods="getUser") public void updateUser() { given().contentType("application/json") .body("{ \"name\": \"John Updated\" }") .when().put("/users/" + userId) .then().statusCode(200); } @Test(priority=4, dependsOnMethods="updateUser") public void deleteUser() { given().when().delete("/users/" + userId).then().statusCode(204); } @Test(priority=5, dependsOnMethods="deleteUser") public void verifyDeleted() { given().when().get("/users/" + userId).then().statusCode(404); } }

"An API endpoint suddenly starts returning different field names in the response. Your tests were passing, now they fail. How do you prevent this from happening?
▼

How I would answer — API contract testing

1
This is an API contract violation. The backend changed the response structure without notifying the frontend/QA team. Field firstName renamed to first_name. Tests that checked jsonPath().getString("firstName") now fail.
2
Prevention — JSON Schema Validation: Add schema validation to every API test. Schema defines required fields and their types. If backend renames a field, schema validation fails IMMEDIATELY — before any other assertions. .body(matchesJsonSchemaInClasspath("userSchema.json"))
3
Prevention — Consumer Driven Contract Testing: Use Pact framework. QA defines what fields they NEED from the API. Any backend change that removes/renames those fields breaks the contract test automatically — even before the change reaches QA environment.
4
Process fix: API versioning (v1/v2 endpoints), breaking change policy, mandatory communication between backend and frontend/QA before merging API changes.
Answer: JSON Schema validation catches structural changes immediately. Add it to every API test. For longer term: consumer-driven contract testing with Pact. Process-wise: API versioning and a no-breaking-changes-without-notice policy.

Common
"How do you decide what to test at the API level vs what to test at the UI (Selenium) level?"
▼

How I would answer — test pyramid thinking

1
API level (more, faster): Business logic validation, data validation (100 different input combinations), error handling (all HTTP error codes), authentication/authorization, performance under load. API tests are 10-100x faster than UI tests.
2
UI level (fewer, critical paths): End-to-end user journeys that span multiple pages. Visual validation (button visible, form layout correct). JavaScript interactions (dropdowns, date pickers). User experience flows that only make sense through the browser.
3
Avoid duplicating tests. If you test "invalid email returns 400" at API level, don't also test it at UI level. UI test just needs to verify the error message APPEARS — not all the business logic behind it.
4
Rule of thumb: Everything you can test at API = test at API. Only use UI tests for what REQUIRES a browser (visual layout, user journeys, JavaScript interactions).
Answer: Test business logic, validation, error handling at API level — it's faster and more reliable. Use Selenium only for end-to-end user journeys and browser-specific interactions. Never duplicate the same test at both levels.

Read the Jenkins console output first. The error message and stack trace tell you exactly what failed and why. Don't guess — read the error.

S31
🔥 Common Live Coding
"Given an employees table (emp_id, name, dept_id, salary, manager_id), write a query to find all employees who earn more than their manager."
▼

How I would answer — think aloud

1
Think first: Both employee and manager are in the SAME table. I need to join the table with itself — a SELF JOIN. Employee's manager_id = Manager's emp_id.
SELECT e.name AS employee_name, e.salary AS employee_salary, m.name AS manager_name, m.salary AS manager_salary FROM employees e JOIN employees m ON e.manager_id = m.emp_id WHERE e.salary > m.salary;
💡 Why JOIN not LEFT JOIN? INNER JOIN excludes employees with no manager (CEO). If we want to include them with NULL manager salary, use LEFT JOIN — but the WHERE condition e.salary > m.salary would filter out NULLs anyway.
Explanation: Self-join aliases the same table as 'e' (employee) and 'm' (manager). Join condition: employee's manager_id = manager's emp_id. Filter: employee salary > manager salary.
S32
🔥 Common Live Coding
"Find departments that have NO employees. Write the SQL query."
▼

How I would answer — 3 ways

-- Method 1: LEFT JOIN + IS NULL (most common) SELECT d.dept_name FROM departments d LEFT JOIN employees e ON d.dept_id = e.dept_id WHERE e.dept_id IS NULL; -- Method 2: NOT IN subquery SELECT dept_name FROM departments WHERE dept_id NOT IN (SELECT DISTINCT dept_id FROM employees WHERE dept_id IS NOT NULL); -- Method 3: NOT EXISTS SELECT dept_name FROM departments d WHERE NOT EXISTS (SELECT 1 FROM employees e WHERE e.dept_id = d.dept_id);
💡 Method 1 is preferred. LEFT JOIN keeps all departments, even with no employees. Where employee columns are NULL = no matching employee = empty department.
Answer: LEFT JOIN departments with employees. Where employee side is NULL = no employees in that department. This is the classic "find rows in A with no match in B" pattern.
S33
🔥 Common
"Find the department with the highest average salary. If two departments tie, show both."
▼

How I would answer

-- Step 1: Calculate avg salary per dept -- Step 2: Find max of those averages -- Step 3: Return all depts with that max avg SELECT d.dept_name, AVG(e.salary) AS avg_salary FROM employees e JOIN departments d ON e.dept_id = d.dept_id GROUP BY d.dept_id, d.dept_name HAVING AVG(e.salary) = ( SELECT MAX(avg_sal) FROM ( SELECT AVG(salary) AS avg_sal FROM employees GROUP BY dept_id ) sub );
💡 Why not just ORDER BY + LIMIT 1? That would miss ties. The subquery approach handles ties correctly by finding the maximum average and returning ALL departments with that average.
Answer: GROUP BY department, calculate AVG salary per group, HAVING clause filters to groups where average equals the maximum average across all departments. Handles ties correctly.

"Your team's automation database has duplicate test results — same test case recorded twice. Write SQL to delete duplicates keeping the latest entry."
▼

How I would answer

-- Table: test_results (id, test_name, status, executed_at) -- Goal: keep only the LATEST execution per test_name -- Step 1: Identify duplicates SELECT test_name, COUNT(*) FROM test_results GROUP BY test_name HAVING COUNT(*) > 1; -- Step 2: Delete keeping only MAX id (latest) DELETE FROM test_results WHERE id NOT IN ( SELECT MAX(id) FROM test_results GROUP BY test_name ); -- Verify result SELECT test_name, COUNT(*) FROM test_results GROUP BY test_name HAVING COUNT(*) > 1; -- Should return 0 rows
Answer: Keep MAX(id) per group = latest entry. Delete everything NOT in that set. Always verify with a duplicate-check query after deletion.

Common
"Write a query to show each employee's salary compared to their department average — above average, below average, or equal."
▼

How I would answer — CASE + Window Functions

SELECT name, dept_id, salary, ROUND(AVG(salary) OVER (PARTITION BY dept_id), 2) AS dept_avg, CASE WHEN salary > AVG(salary) OVER (PARTITION BY dept_id) THEN 'Above Average' WHEN salary < AVG(salary) OVER (PARTITION BY dept_id) THEN 'Below Average' ELSE 'Equal to Average' END AS salary_category FROM employees ORDER BY dept_id, salary DESC;
💡 Window function AVG() OVER (PARTITION BY dept_id) calculates department average for EACH row without collapsing rows like GROUP BY would. This is the power of window functions.
Answer: Window function AVG() OVER PARTITION BY calculates dept average per row. CASE statement categorizes each employee. Elegant — no subquery, no self-join needed.
