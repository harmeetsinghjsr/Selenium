BDD Cucumber — Behavior Driven Development
Introduction to BDD · Gherkin Syntax · Feature Files · Step Definitions · Hooks · Runner · Reports · Integration with Selenium
What is BDD and Why?
💡
BDD = Behavior Driven Development. A software development approach where tests are written in plain English that EVERYONE — developers, testers, and business people — can read and understand. No technical jargon. The test IS the documentation.
Problem BDD Solves
Traditional test code — only developers understand it. Business analyst writes requirement in Word document. Developer interprets it differently. Tester tests something else. Everyone is confused.

BDD fixes this — tests written in language everyone understands.
How BDD Works
1. Business writes plain English requirements
2. These become Feature Files using Gherkin syntax
3. Developer writes Step Definitions (Java code) behind each English line
4. Cucumber runs the feature file, executes the Java code behind each step
Cucumber
The BDD framework for Java. Reads feature files, maps each step to Java method, executes and reports results.
Gherkin
The plain English language used to write feature files. Uses keywords: Feature, Scenario, Given, When, Then, And, But.
Living Documentation
Feature files serve as documentation that is always up to date because if code changes, tests fail — forcing documentation update.

Step Definitions — Complete Implementation
🔗
Step Definition = Java method that maps to a Gherkin step using annotations. The text in the annotation must exactly match (or use regex/Cucumber expressions) the text in the feature file.
LoginSteps.java package stepDefinitions; import io.cucumber.java.en.*; import org.openqa.selenium.WebDriver; import org.openqa.selenium.chrome.ChromeDriver; import org.testng.Assert; import pages.LoginPage; public class LoginSteps { WebDriver driver; LoginPage loginPage; // Maps to: Given the browser is open @Given("the browser is open") public void openBrowser() { driver = new ChromeDriver(); driver.manage().window().maximize(); } // Maps to: And user is on the login page @Given("user is on the login page") public void navigateToLoginPage() { driver.get("https://the-internet.herokuapp.com/login"); loginPage = new LoginPage(driver); } // {string} = Cucumber expression captures value in "quotes" @When("user enters username {string}") public void enterUsername(String username) { loginPage.enterUsername(username); } @When("user enters password {string}") public void enterPassword(String password) { loginPage.enterPassword(password); } @When("user clicks the Login button") public void clickLogin() { loginPage.clickLoginButton(); } @Then("user should see message {string}") public void verifySuccessMessage(String expectedMsg) { String actualMsg = loginPage.getFlashMessage(); Assert.assertTrue(actualMsg.contains(expectedMsg), "Expected: " + expectedMsg + " | Actual: " + actualMsg); } @Then("user should see error {string}") public void verifyErrorMessage(String expectedError) { Assert.assertTrue(loginPage.getFlashMessage().contains(expectedError)); } @Then("the URL should contain {string}") public void verifyUrl(String urlPart) { Assert.assertTrue(driver.getCurrentUrl().contains(urlPart)); } }
Hooks — @Before and @After
🪝
Hooks = code that runs automatically before/after EACH scenario. Like @BeforeMethod/@AfterMethod in TestNG but for Cucumber. Put browser setup here to keep step definitions clean.
Hooks.java package hooks; import io.cucumber.java.*; import org.openqa.selenium.WebDriver; import org.openqa.selenium.chrome.ChromeDriver; import org.openqa.selenium.OutputType; import org.openqa.selenium.TakesScreenshot; public class Hooks { public static WebDriver driver; // Runs BEFORE each scenario @Before public void setUp(Scenario scenario) { driver = new ChromeDriver(); driver.manage().window().maximize(); System.out.println("Starting: " + scenario.getName()); } // Runs AFTER each scenario — takes screenshot on failure! @After public void tearDown(Scenario scenario) { if (scenario.isFailed()) { // Capture and attach screenshot to Cucumber report byte[] screenshot = ((TakesScreenshot) driver) .getScreenshotAs(OutputType.BYTES); scenario.attach(screenshot, "image/png", scenario.getName()); System.out.println("FAILED: " + scenario.getName()); } driver.quit(); } // Tagged hooks — only for @smoke scenarios @Before("@smoke") public void smokeSetup() { System.out.println("Smoke test setup"); } }
Runner Class — Entry Point
🚀
Runner Class = the class you right-click and run. It tells Cucumber WHERE to find feature files, WHERE to find step definitions, WHICH tags to run, and HOW to generate reports.
TestRunner.java package runner; import io.cucumber.testng.AbstractTestNGCucumberTests; import io.cucumber.testng.CucumberOptions; @CucumberOptions( // Where to find .feature files features = "src/test/resources/features", // Package containing Step Definition classes glue = {"stepDefinitions", "hooks"}, // Run only scenarios with @smoke tag tags = "@smoke", // tags = "@smoke and @regression" ← AND // tags = "@smoke or @regression" ← OR // tags = "not @skip" ← NOT // Report plugins plugin = { "pretty", // readable console output "html:target/cucumber-reports/report.html", // HTML report "json:target/cucumber-reports/report.json", // JSON for CI tools "rerun:target/rerun.txt" // rerun failed tests }, // Show step details in console monochrome = true, // Fail if step definitions are missing (strict mode) // dryRun = true ← check mappings without running browser dryRun = false ) public class TestRunner extends AbstractTestNGCucumberTests { // Extend AbstractTestNGCucumberTests for TestNG integration // Or use @RunWith(Cucumber.class) for JUnit }

Viva Q&A — BDD Cucumber
QWhat is BDD?
Behavior Driven Development. A development approach where tests are written in plain English (Gherkin) readable by both technical and non-technical stakeholders. Bridges gap between business requirements and automated tests.
QWhat is Gherkin?
Plain English language used to write feature files. Uses keywords: Feature, Scenario, Given, When, Then, And, But, Background, Scenario Outline, Examples. Files have .feature extension.
QDifference between Scenario and Scenario Outline?
Scenario = single test case with fixed values. Scenario Outline = same scenario repeated with different data from an Examples table. Like @DataProvider in TestNG.
QWhat is Background in Cucumber?
Steps that run before EACH scenario in a feature file. Like a @BeforeMethod scoped to that feature file. Used for common setup like navigating to a page.
QWhat are Hooks in Cucumber?
@Before and @After annotations that run before/after each scenario. Used for browser setup/teardown and taking screenshots on failure. Can be tagged: @Before("@smoke").
QWhat is dryRun in @CucumberOptions?
When dryRun=true, Cucumber checks if all step definitions exist for all feature file steps WITHOUT running the browser. Useful to verify mappings before actual execution.
QWhat is the role of the Runner class?
Entry point for Cucumber execution. Specifies features location, glue (step definitions package), tags to run, and report plugins. Extends AbstractTestNGCucumberTests for TestNG integration.
QHow do you run only smoke tests?
Tag scenarios with @smoke in feature file. In Runner: tags = "@smoke". Can also run from command line: mvn test -Dcucumber.filter.tags="@smoke"

API Fundamentals — What You Must Know
What is an API?
Application Programming Interface. A contract between two software systems on how to communicate. Like a waiter in a restaurant — you (client) give order to waiter (API) who gets it from kitchen (server) and brings back food (response).
What is REST?
Representational State Transfer. An architectural style for APIs using HTTP. Uses JSON/XML for data. Stateless — each request is independent. Uses standard HTTP methods (GET, POST, PUT, DELETE).
Request Components
URL = endpoint address
Method = GET/POST/PUT/DELETE
Headers = metadata (Content-Type, Auth token)
Body = data sent (JSON/XML) — POST/PUT only
Query params = filters in URL (?key=value)
Path params = values in URL (/users/{id})
Response Components
Status Code = result number (200, 404, 500)
Headers = Content-Type, Cache-Control etc
Body = response data in JSON/XML
Response Time = how long it took
MethodActionBody?Idempotent?
GETRetrieve dataNoYes — same result every call
POSTCreate new resourceYesNo — creates new each time

REST Assured — Given / When / Then Pattern
⚡
REST Assured = Java library for API testing. Uses BDD-style Given/When/Then syntax. Given = setup request (headers, body, auth). When = execute HTTP call. Then = validate response (status, body, headers).
Basic REST Assured structure // Static imports at top — needed for given/when/then import static io.restassured.RestAssured.*; import static org.hamcrest.Matchers.*; // Simple GET — verify status and body given() .baseUri("https://api.example.com") .header("Accept", "application/json") .when() .get("/users/1") .then() .statusCode(200) // verify status code .contentType("application/json") // verify content type .body("name", equalTo("John")) // verify field value .body("email", notNullValue()) // verify not null .body("age", greaterThan(18)); // verify numeric condition
All HTTP Methods in REST Assured
REST Assured — all methods // ── GET request ── Response response = given() .queryParam("country", "India") // ?country=India .queryParam("limit", "10") // &limit=10 .when() .get("https://university.hipolabs.com/search") .then() .statusCode(200) .extract().response(); // Path parameter: GET /users/5 given() .pathParam("id", 5) .when() .get("/users/{id}") .then().statusCode(200); // ── POST request ── String requestBody = "{ \"name\": \"John\", \"email\": \"john@test.com\" }"; Response postResponse = given() .contentType("application/json") .body(requestBody) .when() .post("https://reqres.in/api/users") .then() .statusCode(201) .extract().response(); String createdId = postResponse.jsonPath().getString("id"); // extract created ID // ── PUT request (update entire resource) ── given() .contentType("application/json") .body("{ \"name\": \"John Updated\" }") .when() .put("https://reqres.in/api/users/2") .then() .statusCode(200); // ── PATCH request (update partial) ── given() .contentType("application/json") .body("{ \"name\": \"John Patched\" }") .when() .patch("https://reqres.in/api/users/2") .then() .statusCode(200); // ── DELETE request ── given() .when() .delete("https://reqres.in/api/users/2") .then() .statusCode(204); // No content
Extracting Values — JsonPath
Extract from response // Extract single value String name = given().when().get("/users/1") .then().extract().path("name"); // Extract full response object Response res = given().when().get("/users/1").then().extract().response(); String id = res.jsonPath().getString("id"); int age = res.jsonPath().getInt("age"); String email = res.jsonPath().getString("data.email"); // nested int count = res.jsonPath().getInt("universities.size()"); // array size String first = res.jsonPath().getString("universities[0].name"); // first element System.out.println("ID: " + id + " Name: " + name);
Authentication Types
Different auth types in REST Assured // Basic Authentication given() .auth().basic("username", "password") .when().get("/protected").then().statusCode(200); // Bearer Token (most common in modern APIs) given() .header("Authorization", "Bearer eyJhbGci...") .when().get("/profile").then().statusCode(200); // API Key given() .header("x-api-key", "your-api-key") .when().get("/data").then().statusCode(200); // Digest Authentication given() .auth().digest("username", "password") .when().get("/secure").then().statusCode(200);
Request + Response Specification — Reusability
Reusable specifications (RequestSpecification, ResponseSpecification) // Build reusable request spec (avoid repeating base URI, headers) RequestSpecification reqSpec = new RequestSpecBuilder() .setBaseUri("https://api.example.com") .setContentType("application/json") .addHeader("Authorization", "Bearer token") .build(); // Build reusable response spec ResponseSpecification resSpec = new ResponseSpecBuilder() .expectStatusCode(200) .expectContentType("application/json") .build(); // Use specs — no repetition! given().spec(reqSpec) .when().get("/users/1") .then().spec(resSpec).body("name", equalTo("John")); // Overwrite spec for specific case given().spec(reqSpec) .when().get("/missing") .then().statusCode(404); // overrides spec's 200
JSON Schema Validation
📋
JSON Schema = validates the STRUCTURE of JSON response — field names, data types, required fields. Like checking if the response has the right shape, not just the right values.
JSON schema validation import static io.restassured.module.jsv.JsonSchemaValidator.*; // Create schema file: src/test/resources/userSchema.json // { // "$schema": "http://json-schema.org/draft-04/schema#", // "type": "object", // "properties": { // "id": { "type": "integer" }, // "name": { "type": "string" }, // "email": { "type": "string" } // }, // "required": ["id", "name", "email"] // } given().when().get("/users/1") .then() .statusCode(200) .body(matchesJsonSchemaInClasspath("userSchema.json")); // validates structure
Logging in REST Assured
Request and response logging // Log request details given().log().all() // log everything given().log().headers() given().log().body() // Log response details .then().log().all() .then().log().status() .then().log().ifError() // only log on error (best practice) // Write response log to file PrintStream fileLog = new PrintStream(new File("test-results/api.log")); RestAssured.filters(new ResponseLoggingFilter(fileLog));
Viva Q&A — REST Assured
QWhat is REST Assured?
Java library for API testing. Uses BDD-style Given/When/Then syntax. given() = request setup, when() = HTTP call execution, then() = response validation.
QWhat is the difference between PUT and PATCH?
PUT = update ENTIRE resource. If you only send name field in PUT, all other fields become null. PATCH = update PARTIAL resource. Only fields sent are updated, others stay unchanged.
QWhat is the difference between query param and path param?
Path param = part of URL path: /users/{id} → /users/5. Query param = filter appended to URL: /search?country=India&limit=10. Use queryParam() and pathParam() in REST Assured.
QHow do you extract values from response?
Use extract().path("fieldName") for single value, or extract().response() for full Response object, then use response.jsonPath().getString("field").
QWhat is JSON Schema validation?
Validates the STRUCTURE of JSON response — data types, required fields, field names. Uses matchesJsonSchemaInClasspath("schema.json"). Ensures API contract is not broken.
QWhat is RequestSpecification?
Reusable request specification built with RequestSpecBuilder. Stores common settings (base URI, headers, auth token) so they don't need to be repeated in every test.

Postman — API Testing Tool
Collections · Environments · Tests · Data Parameterization · Swagger UI
Postman Key Concepts
What is Postman?
GUI tool for testing APIs manually and automated. No code needed for basic testing. Create requests, send them, inspect responses, write test scripts in JavaScript. Industry standard for API testing.
Collection
A group of related API requests saved together. Like a test suite for APIs. Can have folders inside. Run entire collection at once with Collection Runner. Share with team via export/import.
Environment
Set of variables that change based on environment (dev, staging, prod). Variable: {{baseUrl}}. In dev env: baseUrl = https://dev.api.com. In prod: baseUrl = https://api.com. Switch env → all requests update automatically.
Variables
Global — available across ALL workspaces
Environment — for current environment
Collection — within a collection
Local — only in current request/script
Use: {{variableName}}
Tests Tab — JavaScript Assertions
Postman Tests tab — JavaScript // Verify status code pm.test("Status code is 200", function() { pm.response.to.have.status(200); }); // Verify response body field pm.test("Name is correct", function() { var json = pm.response.json(); pm.expect(json.name).to.equal("John"); }); // Verify response time pm.test("Response time under 500ms", function() { pm.expect(pm.response.responseTime).to.be.below(500); }); // Save value to environment variable (for next request) var id = pm.response.json().id; pm.environment.set("userId", id); // Get environment variable var savedId = pm.environment.get("userId"); // Verify header pm.test("Content-Type is JSON", function() { pm.response.to.have.header("Content-Type", "application/json"); });
Data Parameterization — CSV in Postman
📊
Create a CSV file with test data. In Collection Runner, upload the CSV. Postman runs the request once for EACH row in CSV. Variables from CSV available as {{columnName}} in request URL/body/headers.
CSV file + request body # testdata.csv username,password,expectedStatus tomsmith,SuperSecretPassword!,200 wronguser,wrongpass,401 admin,admin123,200 # In Postman request body (use {{variable}}): { "username": "{{username}}", "password": "{{password}}" } # In Tests tab: pm.test("Status matches", function() { pm.response.to.have.status(parseInt(pm.iterationData.get("expectedStatus"))); });

Git — Version Control System
Commands · Branching · Merging · Conflict Resolution · GitHub Integration
Git Fundamentals
🌿
Git = distributed version control system. Tracks every change to code over time. Multiple developers can work on same project without overwriting each other. Every change is recorded with who made it and when.
Local Repository
Copy of entire repo on YOUR computer. You work here. Commit here. Push to remote when ready.
Remote Repository
Shared copy on GitHub/GitLab/Bitbucket server. Team pushes to and pulls from here.
Branch
Independent line of development. main/master = stable. Feature branch = work in progress. Merge back when done.
All Git Commands from Your Syllabus

Maven Commands
Common Commands

mvn test — run tests

Dependency Scopes
compile — always available (default)
test — only during testing (TestNG) — NOT in main

Jenkins Job Types
Freestyle Project
GUI-based job configuration. Point and click — no scripting needed. Good for simple builds. You set: Source Code (GitHub URL), Build (mvn clean test), Post-build (publish reports, email). Used in your syllabus hands-on.
Pipeline Job
Code-based job using Groovy script (Jenkinsfile). More powerful and flexible. Can be stored in source control alongside code. Used for complex workflows with multiple stages.
Jenkinsfile — Pipeline as Code pipeline { agent any stages { stage('Checkout') { steps { git url: 'https://github.com/org/selenium-project.git', branch: 'main' } } stage('Compile') { steps { sh 'mvn compile' } } stage('Test') { steps { sh 'mvn clean test' } post { always { junit 'target/surefire-reports/*.xml' // publish results publishHTML(target: [reportDir: 'target/extent-reports', reportFiles: 'report.html']) } failure { mail to: 'team@company.com', subject: 'Build FAILED', body: 'Check Jenkins for details' } } } } }

Selenium Grid Architecture
🌐
Selenium Grid = run tests on MULTIPLE browsers and machines SIMULTANEOUSLY. Hub distributes tests to Nodes. Each Node has a browser. Results from all Nodes reported back to Hub.
Hub
Central server. Receives test requests. Distributes to nodes based on browser capability requested. Runs on localhost:4444. All nodes register with Hub. One Hub, many Nodes.
Node
Worker machine with actual browser installed. Registers with Hub. Waits for tests. Executes tests when Hub assigns. Can have multiple browsers on one Node.
Selenium Grid 4 Setup # Terminal 1 — Start Hub java -jar selenium-server-4.x.jar hub # Terminal 2 — Start Node (registers with Hub) java -jar selenium-server-4.x.jar node --hub http://localhost:4444 # Or standalone mode (Hub + Node in one) java -jar selenium-server-4.x.jar standalone
Test code — send test to Grid // Use RemoteWebDriver to send tests to Hub ChromeOptions options = new ChromeOptions(); URL hubUrl = new URL("http://localhost:4444"); WebDriver driver = new RemoteWebDriver(hubUrl, options); // Hub routes to a Node that has Chrome available driver.get("https://example.com"); // test runs on the Node machine, not your local machine

Docker Fundamentals
🐳
"Works on my machine" problem — code works on developer's laptop but breaks on test server. Docker solves this by packaging code + ALL dependencies + configuration into one portable unit called a Container. Runs identically everywhere.
Docker Image
Blueprint/template. Read-only. Contains OS, dependencies, app code. Like a class in Java. Built from Dockerfile. Published to Docker Hub (public registry).
Docker Container
Running instance of an image. Like an object in Java. Isolated, lightweight. Multiple containers from same image. Start, stop, delete without affecting image.
Docker Hub
Public registry for Docker images. Like Maven Central but for Docker images. Pull official images: selenium/hub, selenium/node-chrome, postgres, nginx etc.
VM vs Docker
VM = full OS copy per VM (heavy, 10+ GB, slow to start)
Docker = shares host OS kernel (lightweight, MB size, starts in seconds)
Docker containers = much more efficient than VMs
Why Docker for Testing?
✅ Consistent environment — dev/test/prod same
✅ Selenium Grid in Docker — easy setup
✅ Clean browser for every test run
✅ Run many browser containers in parallel
✅ No "ChromeDriver version mismatch" issues
Docker Commands
Docker commands # Pull image from Docker Hub docker pull selenium/hub docker pull selenium/node-chrome # Run a container docker run -d -p 4444:4444 selenium/hub # -d = detached (background), -p = port mapping (host:container) # List running containers docker ps # List all containers (including stopped) docker ps -a # Stop a container docker stop container-id # Remove container docker rm container-id # List images docker images # Remove image docker rmi selenium/hub # Build custom image from Dockerfile docker build -t my-selenium-image . # View container logs docker logs container-id
Dockerfile — Custom Image
Dockerfile # Start from official Java image FROM openjdk:17-slim # Set working directory inside container WORKDIR /app # Copy pom.xml and src code COPY pom.xml . COPY src ./src # Install Maven and run tests RUN apt-get update && apt-get install -y maven RUN mvn dependency:resolve # Command to run when container starts CMD ["mvn", "clean", "test"]
Selenium Grid with Docker Compose
🔧
Docker Compose = tool to run MULTIPLE containers together with one command. Define all containers in docker-compose.yml file. Run with docker-compose up. Perfect for Selenium Grid — Hub + multiple browser Nodes.
docker-compose.yml — Selenium Grid version: "3" services: # Selenium Hub selenium-hub: image: selenium/hub:4.20.0 ports: - "4444:4444" environment: - SE_NODE_MAX_SESSIONS=5 # Chrome Node — can scale to many instances chrome: image: selenium/node-chrome:4.20.0 depends_on: - selenium-hub environment: - SE_EVENT_BUS_HOST=selenium-hub - SE_EVENT_BUS_PUBLISH_PORT=4442 - SE_EVENT_BUS_SUBSCRIBE_PORT=4443 shm_size: 2gb # shared memory for browser # Firefox Node firefox: image: selenium/node-firefox:4.20.0 depends_on: - selenium-hub environment: - SE_EVENT_BUS_HOST=selenium-hub # Run: docker-compose up -d # Scale chrome: docker-compose up --scale chrome=5
Viva Q&A — Docker
QWhat is Docker?
Containerization platform that packages applications and all their dependencies into lightweight, portable containers. Solves "works on my machine" problem. Containers run identically across dev, test, and production environments.
QDifference between Docker Image and Container?
Image = read-only blueprint/template (like a class). Container = running instance of image (like an object). Multiple containers can run from same image. Deleting a container doesn't affect the image.
QVM vs Docker Container?
VM = runs full OS copy — heavy (GBs), slow to start, resource intensive. Docker = shares host OS kernel — lightweight (MBs), starts in seconds, much more efficient. Docker is preferred for CI/CD.
QWhat is Docker Compose?
Tool for defining and running multi-container Docker applications. One docker-compose.yml file defines all services. docker-compose up starts them all. Perfect for Selenium Grid — Hub + browser Nodes.
QHow does Docker help with Selenium Grid?
Run Selenium Hub and browser Nodes as Docker containers. Clean environment for every test run. Easily scale — run 10 Chrome containers in parallel. No driver version conflicts. Consistent across all machines.

Generative AI Fundamentals
What is Generative AI?
AI that can CREATE new content — text, code, images, audio. Learns patterns from massive training data. Examples: ChatGPT (text), Copilot (code), DALL-E (images), Claude (text). Powered by Large Language Models (LLMs).
How LLMs Work
Trained on billions of text/code samples. Learns patterns and relationships. Given input (prompt), predicts most likely next tokens. Does NOT "understand" — statistically predicts what should come next based on training.
GenAI in QA Testing
✅ Generate test cases from requirements
✅ Create XPath/CSS locators
✅ Write Gherkin feature files
✅ Generate test data
✅ Suggest bug fixes
✅ Review and explain code
✅ Create edge case scenarios
Tokens and Context Window
Token = word or word-piece (~4 chars). LLMs have a context window — max tokens they can process at once. GPT-4 = 128k tokens. Longer context = can read more of your code/requirements. Manage tokens by being concise in prompts.
Prompt Engineering
✍️
Prompt Engineering = the skill of writing clear, specific instructions to AI models to get high-quality, accurate outputs. Poor prompt → generic useless answer. Good prompt → exactly what you need.
Bad Prompt ❌
"Write Selenium test"

Too vague. AI doesn't know: what to test, which page, which framework, what to verify.
Good Prompt ✅
"Write a Selenium Java TestNG test for the login page at https://the-internet.herokuapp.com/login using Page Object Model. Test should: 1) Enter username 'tomsmith', 2) Enter password 'SuperSecretPassword!', 3) Click login, 4) Assert success message contains 'secure area'. Use @FindBy annotations and WebDriverManager."
Prompt Tips
Be specific — mention language, framework
Give context — explain the scenario
Provide examples — show format wanted
Specify constraints — "use TestNG, not JUnit"
Ask for explanation — "explain each line"
Use in BDD
Prompt: "Given this user story: [story], generate a Gherkin feature file with 5 scenarios covering happy path and edge cases." AI generates complete .feature file ready to use.
Use for Locators
Prompt: "Given this HTML: [paste HTML], generate XPath and CSS selectors for the username input field, password field, and submit button." AI gives all locators instantly.
